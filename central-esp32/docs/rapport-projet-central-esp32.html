Rapport technique - Central ESP32 GameRoom

Date : 20 avril 2026

Projet : central-esp32

Objectif : centrale ESP32 pour gérer les coins, les postes de jeu, le Wi-Fi, la sécurité admin et l'interface web locale.

Accès web prévu : http://gameroom.local ou adresse IP locale de l'ESP32.

1. Résumé exécutif

Le projet est une application Arduino/ESP32 organisée en services C++ autour d'un serveur web embarqué. Le rôle de la carte centrale est de recevoir les impulsions d'un monnayeur, convertir ces impulsions en crédits, assigner des crédits aux postes, suivre leur statut, offrir une interface d'administration et permettre une configuration Wi-Fi autonome.

Le code est globalement bien séparé : chaque domaine principal dispose d'un fichier .h et d'un fichier .cpp. Les fichiers les plus volumineux sont WebPage.cpp, qui contient l'interface HTML/CSS/JS, WebRoutes.cpp, qui expose les routes HTTP/API, et WifiProvisioning.cpp, qui gère le portail Wi-Fi.

Pendant ce scan, le code a été légèrement organisé : valeurs par défaut centralisées dans AppConfig.h, suppression d'une allocation dynamique inutile dans WebRoutes.cpp, et conservation de la correction anti-blocage sur la découverte réseau.

3. Architecture du code

central-esp32.ino
  -> initialise stockage, authentification, Wi-Fi, services et serveur web
  -> boucle principale : serveur HTTP, monnayeur, statuts postes, découverte réseau

AppConfig.h
  -> constantes globales : Wi-Fi, mDNS, port, GPIO, timeouts, valeurs par défaut

AppState.h / Post.h
  -> modèles de données en mémoire

Services métier
  -> AuthService : sessions et token API
  -> CoinService : impulsions monnayeur et crédits
  -> PostService : règles métier des postes
  -> PosteClient : client HTTP vers les postes
  -> DiscoveryService : scan réseau progressif
  -> StorageService : sauvegarde Preferences et JSON
  -> LogService : événements internes
  -> WifiService / WifiProvisioning : connexion Wi-Fi, mDNS, portail captif

Web
  -> WebRoutes : endpoints HTTP/API
  -> WebPage : pages HTML embarquées

5. Flux de démarrage

  1. setup() initialise le port série, le stockage et charge AppState.
  2. AuthService::begin() initialise la couche auth.
  3. WifiService::begin() tente une connexion avec les identifiants Wi-Fi sauvegardés.
  4. Si la connexion échoue ou si aucun identifiant n'existe, le mode AP/provisioning démarre.
  5. Si le Wi-Fi fonctionne, mDNS démarre : http://gameroom.local.
  6. Les services monnayeur, postes et routes web démarrent.
  7. loop() sert les requêtes web et exécute les tâches périodiques.

8. Documentation des services

AuthService

Fournit login, logout, vérification cookie et bearer token. Le token API est généré avec esp_random(). La session web est stockée en mémoire : elle disparaît au redémarrage, ce qui est acceptable pour une console admin embarquée.

CoinService

Configure le GPIO AppConfig::COIN_PIN en INPUT_PULLUP et attache une interruption sur front descendant. L'interruption incrémente pulseCount après anti-rebond. update() convertit les impulsions en crédits selon pulsesPerCoin.

PostService

Valide l'unicité des ID/IP, envoie les commandes aux postes, sauvegarde les changements et rafraîchit les statuts. Le rafraîchissement est progressif pour éviter qu'un poste hors ligne bloque toute l'interface.

PosteClient

Utilise HTTPClient pour communiquer avec les postes : POST http://IP/command et GET http://IP/status. Le timeout standard est AppConfig::POSTE_STATUS_TIMEOUT_MS, et la découverte utilise un timeout plus court.

DiscoveryService

Le scan réseau est lancé explicitement par requestScan(). Ensuite run() teste une adresse IP par passage dans la boucle principale. Cette organisation empêche le serveur web de rester bloqué pendant un scan complet du subnet.

WifiService et WifiProvisioning

WifiService gère la connexion normale et mDNS. WifiProvisioning gère le mode AP avec portail captif : page de configuration Wi-Fi, scan des réseaux et sauvegarde des credentials.

9. Organisation appliquée pendant le scan

  • Ajout de constantes par défaut dans AppConfig.h : mot de passe admin, durée par coin, impulsions par coin, crédits initiaux.
  • AppState.h utilise maintenant ces constantes au lieu de valeurs codées en dur.
  • StorageService.cpp utilise les mêmes constantes lors du chargement Preferences.
  • PosteClient.h utilise AppConfig::POSTE_STATUS_TIMEOUT_MS comme timeout par défaut.
  • WebRoutes.cpp remplace new const char*[2] par un tableau statique AUTH_HEADERS.
  • La correction précédente de découverte réseau progressive a été conservée et documentée.

10. Points d'attention

Compilation non vérifiée localement : arduino-cli n'est pas installé dans le shell courant. Il faut compiler depuis l'IDE Arduino ou installer arduino-cli pour une vérification automatisée.
  • Sécurité : le mot de passe par défaut est admin1234. Il doit être changé dès la première installation.
  • mDNS : gameroom.local dépend du support mDNS du client. Sur certains Windows, Bonjour/mDNS peut être nécessaire.
  • Mémoire : les documents JSON de 8192 octets dans WebRoutes.cpp sont corrects pour un petit parc de postes, mais doivent être surveillés si la liste grandit.
  • Interface embarquée : WebPage.cpp contient beaucoup de HTML/JS dans un raw string. C'est simple pour Arduino, mais plus difficile à maintenir qu'un fichier généré.

11. Recommandations

  1. Compiler et uploader après les changements, puis tester http://gameroom.local.
  2. Depuis le dashboard, vérifier que /posts et /logs répondent sans blocage.
  3. Lancer une découverte réseau et confirmer que la page reste responsive pendant le scan.
  4. Changer le mot de passe admin et sauvegarder le token API généré.
  5. Si le projet grandit, extraire progressivement le JavaScript/CSS de WebPage.cpp vers un outil de génération ou des fichiers servis depuis LittleFS/SPIFFS.
